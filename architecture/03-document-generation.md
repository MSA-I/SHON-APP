# SOP 03 — Document Generation (DOCX → PDF)

> **Decision (2026-05-20):** generate `.docx` natively, not PDF. Rationale below. PDF is produced from the DOCX by Word/LibreOffice ("Save as PDF") — a one-click step Shon already knows.

## Rationale

After the L3 POC succeeded with `pdf-lib` + manual bidi reverser, we discovered limitations:
1. No native bidi: every Hebrew string requires manual reversal
2. Mixed RTL+LTR runs (date inside Hebrew sentence) need a custom segmenter
3. Decorative Unicode (`❖`) blocked by WinAnsi encoding limits
4. Editing post-generation is impossible

DOCX solves all four:
1. **Native bidi:** Word handles RTL automatically — set `rightToLeft: true` on runs, `bidirectional: true` on paragraphs, done.
2. **Numbers in Hebrew sentences:** zero special handling. `14.06.2026` reads correctly inside any Hebrew paragraph.
3. **Any Unicode glyph:** Word supports the full Unicode range natively.
4. **Editable:** Shon can adjust wording or fix typos in Word before final PDF export.

## Stack

- **Library:** `docx` v8 (pure JS, no external deps)
- **Font:** Frank Ruhl Libre (set as `font: 'Frank Ruhl Libre'` on every TextRun)
- **Output:** `.docx` saved into the event's folder (see § File Layout)
- **PDF export path:** the user opens the DOCX → File → Save As → PDF, OR (Phase 5) we offer a Tauri "Export PDF" button that shells out to a converter

## API surface (`app/src/lib/docx.ts`)

```typescript
export type DocxBuildInput = {
  client: Client;
  event: Event;
  selections: {
    tableDesigns: ImageSelection[];
    chuppah: ImageSelection[];
  };
  signature: { dataUrl: string; signedAt: number } | null;
  logoSvg: string;        // contents of assets/logo.svg

export async function buildEventDocx(input: DocxBuildInput): Promise<Uint8Array>;
```

The function is pure: data in, bytes out. The caller writes them to disk.

## Color Palette (DOCX hex) — ALWAYS light-theme

DOCX output is **always** light-theme regardless of `meta.theme` (per `claude.md` Behavioral Rule #13). The `meta.theme` flip is an app-only concern — generated documents are deliverables for print and PDF export, which only make sense on a white page with dark text. `buildEventDocx()` MUST NOT branch on theme state.

| Token | Hex | Use |
|---|---|---|
| ink | `0F0E0C` | body text, titles (always dark on light page) |
| cream | `F5F0E8` | background tints (used sparingly in tables) |
| gold | `C9A961` | dividers, decorative ornaments, section labels |
| goldDark | `A88B47` | small labels (less prominent than gold) |

**Page background is always white** (Word's default). Logo selection inside DOCX always uses `assets/logo.svg` (the dark-on-white variant), never `assets/logo-light.svg`. The same rule applies to any future PDF derivation from the DOCX.

## Bidi Pattern (the canonical RTL paragraph)

```typescript
new Paragraph({
  bidirectional: true,
  alignment: AlignmentType.RIGHT,
  children: [
    new TextRun({
      text: 'תאריך: 14.06.2026',
      rightToLeft: true,
      font: 'Frank Ruhl Libre',
      size: 22,           // half-points (so 22 = 11pt)
      color: '0F0E0C',
    }),
  ],
})
```

The combination of `bidirectional: true` (paragraph) + `rightToLeft: true` (run) is the canonical bidi setup. Word resolves the bidi algorithm internally.

## Embedded Images

For images selected by the couple (table designs + chuppah):
1. Read the image from disk via the Tauri FS provider
2. Optionally downscale to ~600px max (faster doc, smaller file)
3. Embed via `new ImageRun({ data: bytes, transformation: { width: 240, height: 160 } })`
4. Group images in a 2- or 3-column table for a magazine-style layout

## Font Embedding

Frank Ruhl Libre is referenced by name only — DOCX does not embed font binaries. To guarantee the look on Shon's machine, install the font into Windows:
- One-time install: drop `FrankRuhlLibre.ttf` into `C:\Windows\Fonts\`
- Or, use a Tauri shell command to install on first launch

If the font is missing on the viewer's machine, Word substitutes another Hebrew font (David Libre or similar). Layout will be slightly different but legible.
